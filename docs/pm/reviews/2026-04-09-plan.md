## Tasks

### Task 1: Skills-first migration (phased)
- **Severity:** High
- **Dimension:** Competitive / LLM Workflow
- **What's wrong:** Entire plugin is built on `commands/*.md`. Anthropic
  merged commands into skills (`code.claude.com/docs/en/skills`);
  backward compat exists with no published end date. The closest
  competitor (`automazeio/ccpm`) migrated off the exact same `/pm:*`
  slash command surface to an Agent Skill. Users who would benefit
  from auto-activation on conversation context ("let's start working"
  → `session-start`) get nothing.
- **Why it matters:** Legacy architecture. Skills unlock auto-trigger,
  supporting files (`${CLAUDE_SKILL_DIR}`-bundled scripts — see Task 4),
  `context: fork` subagent execution, and `disable-model-invocation`
  control that commands don't have. ccpm's migration is an ecosystem
  signal, not a preference.
- **Research says:** *Inward* — Anthropic skills docs explicitly call
  skills the successor and detail the feature delta. *Outward* —
  ccpm's v1 → current migration is a direct A/B test of the same
  architectural choice, run by a project with 100% eval scores.
- **Fix (phased):**
  1. **Phase 1: One-command proof.** Migrate `commands/pm.md` →
     `skills/pm/SKILL.md` first. Lowest risk — it's informational,
     prints a help table, no side effects. Validates that
     `/pm:pm` still resolves to the same surface via plugin namespacing
     (`<plugin>:<skill>`). Ship as 1.0.18 with both `commands/pm.md`
     and `skills/pm/SKILL.md` present so rollback is one deletion.
     Decide on `disable-model-invocation` per skill — for `pm` it
     should default-allow auto-invocation on "what does pm do" / "how
     do I use this plugin" queries.
  2. **Phase 2: `init`.** Migrate `commands/init.md` → `skills/init/`.
     Set `disable-model-invocation: true` — init is a one-shot
     destructive scaffolder, must be explicit. Delete the legacy
     command file only after one full session validates the skill.
  3. **Phase 3: `audit`.** Migrate `commands/audit.md` →
     `skills/audit/SKILL.md`. Long file — good candidate for supporting
     files: move the Shared Findings Menu spec to
     `skills/audit/findings-menu.md` referenced from SKILL.md.
     `disable-model-invocation: true` — audits have real cost. Keep
     the `pm:auditor` subagent reference unchanged (agents are
     separate from skills).
  4. **Phase 4: Session trio together.** Migrate `session-start`,
     `session-update`, `session-end` in one session — they interact
     through the `docs/pm/~pm-session-active` marker and
     `FUTURE.org`/native-task reconciliation. For each, carefully
     craft the `description` so auto-activation is useful without
     false positives: "start a new work session, load context, pick
     a task" should match "let's start working" but not every
     mention of "session". Consider `disable-model-invocation: true`
     for the first iteration, then relax once auto-trigger accuracy
     is proven.
  5. **Phase 5: Dispatcher mirror.** `command-version/pm-commands/`
     targets `~/.claude/` (non-plugin install). Skills work there
     too — `.claude/skills/<name>/SKILL.md`. Rewrite the dispatcher
     `command-version/pm.md` to be a skill too, or leave it as a
     slash-command dispatcher that reads the sibling skill files.
     Decision deferred until Phases 1–4 land and real usage informs
     the choice.
  6. **Phase 6: Deprecate `commands/`.** After all skills ship and
     bake for one external user cycle, remove `commands/*.md` and
     bump to 2.0.0 (breaking only for anyone pinned to
     `claude --plugin-dir` against an old checkout — negligible,
     plugin is unpublished). Keep the `commands/` directory empty
     with a `README.md` pointer to `skills/` until the next major.
- **Effort:** Large (one phase per session, roughly 6 sessions).
- **Critical constraint:** User-facing slash command names MUST NOT
  change. Plugin skills resolve to `/pm:<name>` via
  `<plugin>:<skill-name>` namespacing, so `/pm:audit`, `/pm:init`,
  etc. remain identical to the user. Per memory rule: ask before
  any user-facing command rename. This migration does not rename
  anything.

### Task 4: Bundle deterministic scripts for session-end hygiene
- **Severity:** Medium
- **Dimension:** Competitive Gap
- **What's wrong:** `/pm:session-end` Phase 1a/1e (uncommitted state
  check, broken-ref scan, CLAUDE.md line count, NOT_IMPLEMENTED grep,
  tracker consistency diff, stale daily-log check) is pure LLM work.
  Every session-end burns tokens on operations that are deterministic
  and instant in bash.
- **Why it matters:** Token cost at every session close — the most
  frequent `/pm` subcommand — and slower than it needs to be.
- **Research says:** `automazeio/ccpm` bundles 14 bash scripts
  (`standup.sh`, `status.sh`, `search.sh`, `next.sh`, `blocked.sh`,
  `epic-list.sh`, ...) as part of its skill, run for status and
  standup queries at zero LLM cost and instant output. README:
  *"Deterministic ops run as scripts — status, standup, search,
  validate all run as bash scripts: fast, consistent, no LLM token
  cost."* This is complementary to Task 1 — skills can bundle scripts
  via `${CLAUDE_SKILL_DIR}`, so this task benefits directly from the
  migration.
- **Fix:**
  1. Create `skills/session-end/scripts/drift-scan.sh` (once Task 1
     Phase 4 lands). Emits JSON: `{"broken_refs": [...], "claude_md_lines":
     N, "not_implemented_matches": [...], "tracker_conflicts": [...],
     "stale_daily_log": bool}`.
  2. `session-end` SKILL.md runs the script first, and the LLM only
     re-analyzes findings the script surfaces. Empty output → skip
     drift scan entirely.
  3. Same pattern for `drift-scan.sh` is reusable from
     `skills/audit/scripts/` for Documents-mode Phase 1 VALIDITY
     checks.
  4. Dispatcher mirror can ship the same scripts — bash works
     uniformly in both install modes.
- **Effort:** Medium (depends on Task 1 Phase 4 for the skills scaffold)
