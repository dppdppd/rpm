# PM Plan — 2026-04-10

## Context

Third full `/rpm:audit project` run on the plugin's own repo. First
audit after the `pm` → `rpm` rename (commits `d443ba6`, `3f8b340`,
`83020ea`) and the hook-driven session lifecycle (2.2.0, `c53f699`).

**Phase 1 (Investigate)** — read all skills, hooks, agents, scan.sh,
CLAUDE.md, RPM.md, PRESENT.md, FUTURE.org, RPM-LOG.md, prior reviews,
daily logs, memory files. Launched `rpm:auditor` in background.
Critical discovery: the `docs/pm/` → `docs/rpm/` rename in `83020ea`
was not propagated to 4 of 5 hook scripts.

**Phase 2 (Inward Research)** — validated against Claude Code docs:

1. **Hooks specification** — hooks.json format is correct. All 5 hook
   events are valid. Discovered `once: true` handler option (v2.1.0)
   that replaces manual flag files. Discovered `$CLAUDE_PLUGIN_DATA`
   (v2.1.78) for persistent plugin state.
2. **Agent frontmatter schema** — 16 supported fields including
   `color` (valid, contrary to the 2026-04-09 audit finding that
   removed it). Plugin-shipped agents don't support `hooks`,
   `mcpServers`, or `permissionMode` (silently ignored).

**Phase 3 (Outward Research)** — 5 competitors analyzed:

1. **ccpm** — spec-driven PM with 14 bash scripts. Complementary:
   planning-to-execution focus, no doc drift detection.
2. **cc-spex** — closest competitor for drift detection. `/spex:evolve`
   for spec/code reconciliation with 95% compliance threshold. Narrower
   scope (specs only).
3. **claude-mem** — infrastructure-layer memory (SQLite + Chroma).
   Complementary: raw recall, no structured project health.
4. **gstack** — 23 specialist commands. Complementary: breadth, no doc
   audit or hooks.
5. **flow-next** — plan-first task orchestration with dependency graphs.
   Complementary: planning focus, no doc alignment.

No competitor combines hook-driven automation, doc drift scoring,
session lifecycle, and task tracking. rpm's niche is defensible.

## Tasks

### Task 1: Fix hook paths — `docs/pm` → `docs/rpm`
- **Severity:** Critical
- **Dimension:** Risk / LLM Workflow
- **What's wrong:** 4 of 5 hooks set `PM_DIR="$PROJECT_DIR/docs/pm"`.
  The directory was renamed to `docs/rpm/` in commit `83020ea`. This
  means ALL hook-driven automation silently fails: SessionStart exits
  early with "no docs/rpm/ directory" (checking wrong path),
  PreCompact/Stop/UserPromptSubmit exit silently (no PM dir).
  Only `post-compact.sh` was correctly updated.
- **Why it matters:** Hook-driven session lifecycle is the plugin's
  core differentiator — claimed in README, CLAUDE.md, skill
  descriptions. It's completely broken.
- **Research says:** Inward — hooks.json format is correct; the bug is
  in the bash scripts' path constants only. This session's own
  SessionStart output confirms: "This project has no docs/rpm/
  directory."
- **Fix:** In these 4 files, change `PM_DIR="$PROJECT_DIR/docs/pm"` to
  `PM_DIR="$PROJECT_DIR/docs/rpm"`:
  - `hooks/session-start-auto.sh:7`
  - `hooks/pre-compact.sh:11`
  - `hooks/stop-learn-capture.sh:7`
  - `hooks/prompt-session-nudge.sh:7`
- **Effort:** Small

### Task 2: Fix project-mode.md WebFetch contradiction
- **Severity:** High
- **Dimension:** LLM Workflow (CONTRADICTORY)
- **What's wrong:** `skills/audit/project-mode.md` lines 57 and 77
  instruct using `WebFetch`. CLAUDE.md lines 67–70 explicitly forbid
  it: "Do not use WebFetch — it has no tool-level timeout and a hung
  request stalls the entire session." The deep-research skill already
  correctly uses `curl` instead.
- **Why it matters:** `/rpm:audit project` Phase 2 and Phase 3 may
  use WebFetch per the skill body, stalling the session. The
  contradiction means the LLM gets conflicting instructions.
- **Research says:** CLAUDE.md references anthropics/claude-code#34565
  as the reason. The deep-research skill is the correct model
  (`allowed-tools` includes `Bash(curl:*)`, not `WebFetch`).
- **Fix:** Replace both WebFetch references in project-mode.md:
  - Line 57: "Use WebSearch / WebFetch directly" → "Use WebSearch /
    `curl --max-time 30 -sL` directly"
  - Line 77: "actually `WebFetch` their docs" → "actually fetch their
    docs (via `curl --max-time 30 -sL`)"
- **Effort:** Small

### Task 6: Adopt `once: true` for nudge hook
- **Severity:** Low
- **Dimension:** Competitive Gap
- **What's wrong:** `prompt-session-nudge.sh` creates a
  `~rpm-nudge-flags/wrapup` file to prevent re-firing the 90-minute
  nudge. Claude Code v2.1.0 added `"once": true` as a hook handler
  option that does this automatically.
- **Why it matters:** Eliminates a manual flag file, its creation
  directory, and its cleanup in session-end Phase 5. Fewer moving
  parts.
- **Research says:** Inward — `once: true` is a supported hook handler
  field. When set, the hook fires at most once per session. This is
  exactly the nudge behavior.
- **Fix:**
  1. Add `"once": true` to the UserPromptSubmit hook handler in
     `hooks/hooks.json`.
  2. Remove the flag-file logic from `prompt-session-nudge.sh`
     (lines 22–25: FLAG creation + early exit).
  3. Remove `docs/rpm/~rpm-nudge-flags` from session-end Phase 5
     cleanup (`rm -rf` command in SKILL.md).
- **Effort:** Small

### Task 7: Extend broken_refs to bash scripts
- **Severity:** Low
- **Dimension:** LLM Workflow
- **What's wrong:** scan.sh `broken_refs` section only checks
  backticked path references in 3 markdown files (CLAUDE.md,
  README.md, RPM.md). Bash scripts that reference project paths
  (like `docs/pm/` in the hooks) are invisible to the scanner.
  This is how the rename bug (Task 1) shipped undetected.
- **Why it matters:** The drift scanner is the plugin's primary
  defense against documentation misalignment. A blind spot in bash
  scripts means code-level path references can drift without
  detection.
- **Research says:** cc-spex has formal spec/code drift reconciliation.
  rpm's scanner could adopt a similar approach for bash paths.
- **Fix:** Add a new scan section or extend `broken_refs` to grep
  bash scripts (`hooks/*.sh`, `skills/*/scripts/*.sh`) for string
  patterns matching `docs/rpm/` and verify the referenced paths
  exist. Use a simpler heuristic than the markdown scanner: extract
  strings containing `docs/` from variable assignments and path
  arguments, check if they resolve.
- **Effort:** Medium

## What's Working (don't break these)

- **CLAUDE.md is 70 lines.** Well under the 150-line budget.
- **scan.sh bundle pattern is solid.** Zero-token drift detection
  with key=value output, auto-injected before skill bodies. The
  pattern works — just needs the path fix.
- **Session-end 5-phase workflow is well-designed.** Auto-apply of
  past/present/future without approval, followed by an action menu
  for everything else. The latency optimizations (parallel reads,
  merged responses) are good.
- **deep-research skill correctly uses curl.** The curl pattern with
  `--max-time 60` and `head -c 100000` is right. Don't regress to
  WebFetch.
- **Competitive positioning is strong.** No tool combines all 5 of
  rpm's focus areas (doc drift, session lifecycle, hook automation,
  audit scoring, task tracking). The documentation alignment niche
  is unique and defensible.
- **Hook architecture is correct.** hooks.json format, event names,
  and handler patterns all match the current spec. The bug is only
  in bash path constants, not in the hook infrastructure.
