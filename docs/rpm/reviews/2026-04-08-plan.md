## Tasks

### Task 1: Fix README install instructions
- **Severity:** High
- **Dimension:** LLM Workflow / User-facing docs
- **What's wrong:** `README.md:23-25` shows only
  `claude --plugin-dir /path/to/claude-plugin-pm`, which is the dev /
  session-scoped install mode, not how users install plugins normally.
- **Why it matters:** First-run UX is wrong. Anyone trying to adopt this
  plugin from the README gets an ephemeral install and no path to a
  persistent one. The README is the front door.
- **Research says:** Per `code.claude.com/docs/en/plugins-reference` and
  `…/discover-plugins`, the canonical flow is
  `/plugin marketplace add <src>` followed by
  `/plugin install pm@dppdppd-plugins`, or `claude plugin install pm@dppdppd-plugins` from CLI.
- **Fix:**
  1. Rewrite the `## Installation` section to lead with a "Dev / local"
     block (current `claude --plugin-dir` form).
  2. Add a "Marketplace install (when published)" block with the
     `/plugin marketplace add …` → `/plugin install pm@dppdppd-plugins`
     flow and a TODO note to fill in the repo URL on publish.
- **Effort:** Small

### Task 2: Wire `/pm:audit` to the `pm:auditor` subagent
- **Severity:** High
- **Dimension:** Architecture
- **What's wrong:** `commands/audit.md` Mode: Standard Phase 1 says
  "Launch a background agent to scan" and inlines a ~50-line scan prompt,
  but never specifies `subagent_type: "pm:auditor"`. Meanwhile
  `agents/auditor.md` holds a near-duplicate spec as a named plugin
  subagent (registered as `pm:auditor`). The two prompts have already
  drifted: auditor.md has 8 short bullets, audit.md has 8 detailed bullets.
- **Why it matters:** Two sources of truth for the same scan logic, only
  one of which runs. Any future edit touches one and silently leaves the
  other stale. The named agent is effectively dead code.
- **Research says:** Per `code.claude.com/docs/en/sub-agents`, plugin-
  shipped subagents are addressed via `subagent_type: "<plugin>:<agent>"`.
  Confirmed in the agent registry for this very session.
- **Fix:**
  1. In `commands/audit.md` Mode: Standard Phase 1, replace the inline
     prompt block with: "Launch the `pm:auditor` subagent (background).
     It will return a structured report. Wait for completion, then score
     findings per Phase 2."
  2. Update `agents/auditor.md` body so it carries the *full* scan spec
     (promote the detailed 8-step prompt from audit.md into auditor.md).
  3. Mirror to `command-version/pm-commands/audit.md` — but note that
     command-version has no subagent mechanism (it's the dispatcher
     install). Leave the inline prompt there, OR simplify it to a
     cross-reference. Flag for the user.
- **Effort:** Medium (requires testing that the rewired command-version
  variant still works end-to-end).

### Task 3: Delete root-level `plugin.json` mirror
- **Severity:** Medium
- **Dimension:** Architecture / Workflow
- **What's wrong:** `/plugin.json` and `/.claude-plugin/plugin.json`
  both exist with identical content. `CLAUDE.md:27-28` requires bumping
  `version` in both. Per the docs, `.claude-plugin/plugin.json` is the
  canonical location and a root-level file is not part of the schema.
- **Why it matters:** Pure maintenance burden: two files to keep in sync,
  drift risk. CLAUDE.md instruction load is higher than it needs to be.
- **Research says:** `code.claude.com/docs/en/plugins-reference` §Plugin
  manifest schema explicitly states "The `.claude-plugin/plugin.json`
  file defines your plugin's metadata and configuration."
- **Fix:**
  1. `rm plugin.json` (repo root).
  2. Edit `CLAUDE.md` Layout and Editing sections: remove the "mirrored
     in `.claude-plugin/plugin.json`" line and the "bump version in both"
     instruction. Simplify to: "Plugin manifest: `.claude-plugin/plugin.json`.
     Bump `version` after meaningful changes."
  3. Verify `command-version/` doesn't reference the root manifest
     (it shouldn't — command-version is a different install shape).
- **Effort:** Small. **Approved by user.**

### Task 6: Add `/pm:session-update` mid-session checkpoint command
- **Severity:** Medium
- **Dimension:** Competitive
- **What's wrong:** The session lifecycle only has start and end. Long
  sessions drift between the two without explicit bookmarks, and
  `/pm:session-end` must reconstruct everything from the conversation.
- **Why it matters:** State loss in long sessions. Mid-session `/clear`
  is unrecoverable without a checkpoint. Hard to diff "what's happened
  since I last paused".
- **Research says:** iannuttall's `claude-sessions` plugin ships
  `/session-update` for exactly this purpose — "appends timestamped
  progress; auto-generates summary if no notes provided". Competitive
  gap with a known, working pattern.
- **Fix:**
  1. Create `commands/session-update.md` — three-phase lightweight
     command: analyze conversation since last checkpoint, append
     `### HH:MM update` subsection to `docs/pm/past/YYYY-MM-DD.md`,
     refresh `PRESENT.md` fields, print one-line confirmation. No menu,
     no commit, no `FUTURE.org` touch, no handoff.
  2. Mirror to `command-version/pm-commands/session-update.md`.
  3. Add to command tables in `commands/pm.md`, `command-version/pm.md`,
     `README.md`, `CLAUDE.md`, `docs/pm/PM.md`.
- **Effort:** Small

### Task 7: Integrate Claude Code native `TaskCreate/TaskUpdate/TaskList` with FUTURE.org
- **Severity:** Medium
- **Dimension:** Competitive
- **What's wrong:** Since Jan 2025 Claude Code ships native persistent
  tasks (`~/.claude/tasks/`) with dependencies, blockers, activeForm,
  and multi-session collaboration. We don't use them. `FUTURE.org` is
  a hand-edited org-mode file duplicating a subset of that
  functionality with worse in-session UX.
- **Why it matters:** Missing free native functionality. Users who
  prefer the native task UI get nothing from our tracker; users who
  prefer `FUTURE.org` can't benefit from native persistence across
  sessions.
- **Research says:** Anthropic upgraded task management Jan 2025 with
  proper multi-session persistence (deps, blockers, spinner
  `activeForm`). Several community plugins already reconcile
  repo-local trackers with native tasks.
- **Fix:** Hybrid approach — `FUTURE.org` remains source of truth
  (versioned in repo), native tasks are the in-session working set.
  1. `commands/session-start.md`: after stating the plan, `TaskCreate`
     a native task for the picked `FUTURE.org` item (subject = heading,
     description = body, activeForm for spinner).
  2. `commands/session-end.md` Phase 2: read `TaskList` and reconcile —
     mark `FUTURE.org` entries DONE for completed native tasks; append
     new TODO / IN-PROGRESS entries for native tasks without a
     counterpart; do NOT delete native tasks (they persist for next
     session).
  3. Mirror to `command-version/pm-commands/`.
- **Effort:** Medium

### Task 8: Restructure `/pm:audit` Heavy mode with explicit outward research phase
- **Severity:** High
- **Dimension:** Process
- **What's wrong:** Heavy mode Phase 2 was titled "Research" with no
  distinction between inward (validate against authoritative docs)
  and outward (competitive analysis). A Claude running Heavy mode
  (like this very session earlier) naturally defaults to inward
  research and misses whether the project's *spec itself* is
  best-in-class. The user had to call it out manually.
- **Why it matters:** Audit findings over-index on spec conformance
  and systematically under-represent competitive gaps. The single
  biggest failure mode of this audit was exactly this.
- **Research says:** N/A — meta-finding about the audit command
  itself. The inward/outward split is standard in consulting
  frameworks (SWOT uses the same distinction).
- **Fix:** Split Heavy mode into 5 phases:
  1. Phase 1: Investigate (unchanged)
  2. Phase 2: **Inward Research** — validate against authoritative
     docs; min 2 dimensions
  3. Phase 3: **Outward Research** — discover 3-5 competitors;
     WebFetch at least 2; identify what they do better and what we
     do better; **required**, not optional
  4. Phase 4: Analyze (was Phase 3) — every finding must cite Phase 1
     evidence AND either Phase 2 or Phase 3 research; add
     "Competitive Gaps" as a 7th analysis dimension
  5. Phase 5: Ask questions and refine (was Phase 4)
  6. Plan file template: add `Competitive` to the Dimension enum
  7. Mirror to `command-version/pm-commands/audit.md`.
- **Effort:** Medium
