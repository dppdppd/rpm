## Phase 2: Inward Research

### Hooks specification

- hooks.json wrapper format (`{"description":..., "hooks":{...}}`)
  is correct. Auto-discovered from `hooks/hooks.json`.
- All 5 events used (SessionStart, PreCompact, PostCompact, Stop,
  UserPromptSubmit) are valid and supported.
- **New feature:** `once: true` handler option (v2.1.0) — fires the
  hook at most once per session. Directly applicable to the nudge
  hook, replacing the manual `~rpm-nudge-flags/wrapup` flag system.
- **New feature:** `$CLAUDE_PLUGIN_DATA` (v2.1.78) — persistent
  state directory for plugins. Could replace `docs/rpm/~rpm-*`
  ephemeral files. Large migration effort; deferred.
- Hook output over 50K chars gets truncated to disk with file path +
  preview — not an issue for rpm hooks (all under 5K).

### Agent frontmatter schema

- 16 supported fields: `name`, `description`, `model`, `color`,
  `tools`, `disallowedTools`, `permissionMode`, `maxTurns`, `effort`,
  `skills`, `mcpServers`, `hooks`, `memory`, `background`,
  `isolation`, `initialPrompt`.
- `color` IS a valid field (contrary to the 2026-04-09 audit finding
  A5 that removed it from `auditor.md`). However, `color` is
  cosmetic only.
- Plugin-shipped agents don't support `hooks`, `mcpServers`, or
  `permissionMode` (silently ignored for security).

## Phase 4: Analysis

### 1. Process Health

The rename workflow (3 commits: `d443ba6` → `3f8b340` → `83020ea`)
had no mechanical verification step for bash scripts. The broken_refs
scanner only checks markdown. FUTURE.org had a "dogfood hooks" TODO
that was not completed before the rename was declared done.

### 2. Architecture & Code Health

Hook architecture is correct — hooks.json format, event names, and
handler patterns all match the current Claude Code spec. The
scan.sh bundle pattern (zero-token drift detection, key=value output,
auto-inject) is well-designed and should be preserved.

### 3. LLM Workflow

- CLAUDE.md at 70 lines is well within the 150-line budget.
- project-mode.md contradicts CLAUDE.md on WebFetch — the deep-research
  skill has the correct `curl` pattern.
- CLAUDE.md and RPM.md have stale namespace references from the rename.
- scan.sh `pm_docs_staleness` references the wrong directory.

### 4. Risk & Compliance

**Critical risk:** The plugin's core differentiator (hook-driven
session lifecycle) is completely non-functional. All marketing claims
(README, CLAUDE.md, skills) describe automation that doesn't work.
This is the highest-priority fix.

### 5. Strategic Direction

The documentation alignment positioning is strong. The competitive
research confirms no tool occupies this niche. The README rewrite
(uncommitted) correctly frames the problem and positioning.

### 6. Session Discipline

FUTURE.org is well-maintained. One open TODO ("Dogfood rpm hooks")
represents the exact gap that caused the hooks bug. Session discipline
would have caught this had the dogfood step been completed before
declaring the rename done.

### 7. Competitive Gaps

- cc-spex's formal compliance scoring (95% threshold) is more
  structured than rpm's audit scoring. Consider adopting a threshold
  model for doc health.
- `once: true` from Claude Code v2.1.0 eliminates manual flag files.
  Direct improvement.
- `$CLAUDE_PLUGIN_DATA` for ephemeral state is a clean separation but
  large migration effort. Deferred.
