### Task 6: Adopt context-pressure monitoring pattern
- **Severity:** Medium
- **Dimension:** Competitive Gap
- **What's wrong:** rpm currently has no token-count-aware signal
  for "you're running out of context — wrap up." Session 3 removed
  the time-based 90-minute nudge in favor of context-aware model
  judgment (`/session-end suggestion via context.md`). That is
  strictly smarter than time-based but still relies on model
  judgment, not on actual transcript size.
  
  The direct competitor `claude-code-session-kit` uses a
  **PostToolUse hook** (matcher `.*`, every 10 tool calls) that
  reads `transcript_path` from stdin JSON, runs `wc -c` on the
  transcript file, and injects a context warning at three
  thresholds (400KB warn / 600KB alert / 700KB hard stop — roughly
  40/60/70% of a 1MB-context run). The hard stop uses
  `hookSpecificOutput.decision.additionalContext` to force the
  model into handoff mode.
  
  This pattern is Claude-Code-native, zero-SDK, 50 lines of bash,
  and "battle-tested across 92 real sessions." It would pair well
  with rpm's already-present session-end workflow: warnings at
  40/60% would hint "run /session-end soon"; hard stop at 70%
  would force it.
- **Why it matters:** Every rpm user will eventually hit the
  context ceiling mid-task. The existing PreCompact checkpoint
  rescues state on compaction, but by that point valuable detail
  is already being summarized away. Catching context pressure
  earlier (40%/60%) lets the user commit + hand off cleanly
  before compaction kicks in.
- **Research says:** Inward — Claude Code's PostToolUse hook
  accepts `hookSpecificOutput.decision.additionalContext`;
  `transcript_path` is part of the PostToolUse stdin schema. Both
  confirmed against code.claude.com docs.
  Outward — shihchengwei-lab/claude-code-session-kit
  context-lifecycle/context-monitor.sh, fetched end-to-end;
  implementation pattern validated.
- **Fix:**
  1. Add `plugin/hooks/context-monitor.sh` — PostToolUse hook that
     maintains a call counter in `/tmp` (skip first 3, then every
     10th), reads `transcript_path`, `wc -c`s it, emits
     `additionalContext` at three thresholds.
  2. Register under `PostToolUse` in `hooks/hooks.json` with
     matcher `".*"`.
  3. Wire the "soft" thresholds to suggest `/session-end`; wire the
     hard threshold to require it (model should refuse to start new
     tasks until the session is wrapped).
  4. Document the thresholds in plugin/CLAUDE.md's hooks section.
- **Effort:** Medium

### Task 8: Adopt SessionEnd hook for wrap-up enforcement
- **Severity:** Medium
- **Dimension:** Competitive Gap (Inward)
- **What's wrong:** Claude Code has had a `SessionEnd` event since
  the last audit (fires on `clear`, `resume`, `logout`,
  `prompt_input_exit`, `bypass_permissions_disabled`, `other`). rpm
  does not register it. This means ending a session via `/exit`
  (`prompt_input_exit`) or by closing the client (`logout`) leaves
  no rpm-side record — exactly what happened in Session 4 (no
  `/session-end` run, two files left uncommitted). Today's audit
  surfaced this gap from its own consequences.
- **Why it matters:** The `/session-end` skill is user-opt-in. A
  `SessionEnd` hook can:
  1. Detect unwrapped sessions (marker file still present →
     session ended without `/session-end`).
  2. Write a minimal terminal-visible warning to stderr: "rpm:
     session ended without wrap-up; uncommitted: N files".
  3. Optionally write a stub daily-log entry so the gap is
     documented even if the user didn't run `/session-end`.
- **Research says:** Inward — code.claude.com/hooks.md
  documents `SessionEnd` schema (matcher by termination reason).
  Outward — claude-code-session-kit doesn't use SessionEnd but does
  use Stop hook for similar validation (see Task 9 below).
- **Fix:**
  1. Add `plugin/hooks/session-end-hook.sh` (naming to avoid
     confusion with the `session-end` skill): if marker still
     exists, print stderr warning + append stub to
     `docs/rpm/past/$(date +%Y-%m-%d).md` noting the unwrapped
     session ID and session summary (from `compact_summary` if
     present).
  2. Register under `SessionEnd` in `hooks/hooks.json`.
  3. Document in plugin/CLAUDE.md and plugin/README.md.
- **Effort:** Medium

## Deferred / Not Recommended

- **`$CLAUDE_PLUGIN_DATA` migration.** Moving `docs/rpm/~rpm-*`
  files into the plugin data directory would be cleaner by Claude
  Code plugin convention, but rpm deliberately keeps state visible
  (in git, in the project tree, under rpm's own paths). Transparency
  > convention here. Revisit only if a user reports a conflict.
- **Harness-agnostic / agentskills.io adoption.** ccpm went this
  route. rpm's hook-driven automation is Claude-Code-specific by
  design; adopting the agnostic standard would require dropping the
  core differentiator. Stay Claude-Code-native.
- **Tiered CLAUDE.md templates (session-kit pattern).** Shipping
  3 templates (beginner/intermediate/advanced) adds surface area
  for questionable win. rpm's single template already scales from
  solo to medium teams. Skip.
- **Marketplace submission to claude-plugins-official.** Worth
  doing, but not until 2.4.0 is cut and context-pressure monitoring
  (Task 6) or SessionEnd hook (Task 8) lands. Post-2.4.0 todo.
