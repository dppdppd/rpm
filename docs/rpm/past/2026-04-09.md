### Shipped 2.1.1 — session-start PRESENT.md drift check (`222eb03`)

The reconcile above surfaced a systematic gap: commits that land
outside `/pm:session-end` (crash, `/clear` without session-end,
forgot, etc.) never trigger a PRESENT.md update, because
session-end Phase 2 only updates "fields that actually changed
this session." 1.0.23 and 2.0.0 both slipped through.

Fix is trivial — one git command in session-start Phase 2:

```bash
LAST=$(git log -1 --format=%H -- docs/pm/PRESENT.md)
[ -n "$LAST" ] && git log --oneline "${LAST}..HEAD"
```

If output is non-empty, session-start surfaces the commit list to
the user and asks whether to reconcile before picking a task.
Five lines of skill body + one `Bash(git log:*)` frontmatter entry.
No new scripts, no cross-skill references. Scope-larger options
(shared scan library, new scan.sh section, etc.) rejected as
over-engineered for a single git invocation.

Self-healing: the exact drift that forced today's reconcile will
self-detect next time.

## Key discoveries

- **Session-end's "update only changed fields" creates systematic
  drift-slip.** When a commit lands outside a session-end flow, the
  PRESENT.md version bump + Completed Work entry never happens.
  Drift compounds silently until a reconciliation session. Now
  closed by the 2.1.1 session-start catch-up check.

- **Volta's spec layout is `docs/spec/` (singular) + `docs/spec/
  traits/`.** Fix-logs live at `docs/parity-fix-log.md` (top-level
  `docs/`), not under `docs/pm/`. First scan.sh iteration assumed
  `specs/` top-level and missed volta entirely. Lesson: check real
  target layouts before writing heuristic path guards. Widened to
  cover four common spec locations + both `docs/` and `docs/pm/`
  for staleness.

- **Cross-skill script references via `${CLAUDE_PLUGIN_ROOT}` work
  in skill bodies.** `hooks/hooks.json` already used the variable,
  but no skill body had. The `/pm:audit quick` target is the first.
  Pattern: `!${CLAUDE_PLUGIN_ROOT}/skills/<other-skill>/scripts/
  <script>.sh`. Unlocks script-bundle reuse across skills without
  duplication or symlinks.

- **Commands-era audit cycle friction ≠ skills-era audit friction.**
  Volta's 4-cycles-in-5-days pattern was a commands-era artifact
  (pre-`disable-model-invocation`, audit was easily auto-invoked).
  Skills-era audit is gated and scoped by its description alone.
  Per-run cost (110-doc walk) is unchanged, but invocation cadence
  should drop without further intervention.

## What didn't work

- **First scan.sh iteration used `[ -d specs ]` guard.** Plugin has
  no `specs/` so it skipped silently in local dogfooding. Then
  tested against volta and realized volta's specs are at `docs/
  spec/` — the check skipped volta too. Had to generalize the find
  path list and guard on empty result, not dir existence. Caught
  because I explicitly cross-tested against volta instead of
  trusting the plugin-only dogfood.

- **First status label was `no_specs_or_present` (combined).**
  Misleading — suggested "neither exists" when the actual condition
  was usually "no specs dir, PRESENT.md fine". Refined to two
  distinct branches (`no_spec_dir` vs `no_present_md`) for clarity.

- **`/pm:session-end` skill hit a permission prompt on auto-inject
  scan.sh.** The `!\`${CLAUDE_PLUGIN_ROOT}/skills/session-end/
  scripts/scan.sh\`` pattern inside the skill body triggered a
  shell-command permission check that the skill tool couldn't
  auto-resolve. Worked around by invoking scan.sh via the Bash
  tool directly. Worth investigating whether `allowed-tools` needs
  a specific `Bash(…)` pattern for auto-inject, or whether this is
  an environment-specific permission policy. Affects the new
  `/pm:audit quick` target too — same auto-inject pattern.
