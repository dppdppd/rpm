## Phase 2: Inward Research

Validated against `code.claude.com/docs`. Two dimensions, both
changed the assessment materially.

### Dim 1 — Skills as the successor to `.claude/commands/`

**Sources:**
- `code.claude.com/docs/en/skills` (fetched)
- `code.claude.com/docs/en/plugins-reference` (fetched)
- `anthropics/claude-code#37447` (gh issue view, OPEN, stale-labeled)

**Key quote, verbatim from the skills doc:**

> Custom commands have been merged into skills. A file at
> `.claude/commands/deploy.md` and a skill at
> `.claude/skills/deploy/SKILL.md` both create `/deploy` and work
> the same way. Your existing `.claude/commands/` files keep
> working. Skills add optional features: a directory for supporting
> files, frontmatter to control whether you or Claude invokes them,
> and the ability for Claude to load them automatically when
> relevant.

**Skill feature delta over commands:**
- `description`-based auto-activation on matching conversation context
- Supporting-file directory; `${CLAUDE_SKILL_DIR}` substitution
- `disable-model-invocation: true` (user-only)
- `user-invocable: false` (Claude-only background knowledge)
- `allowed-tools` granted without permission prompts while skill
  is active
- `paths` glob-based automatic activation
- `context: fork` with `agent: Explore|Plan|general-purpose|<custom>`
  for subagent execution of the skill content
- `hooks` scoped to the skill's lifecycle
- `effort` level override per skill
- `$ARGUMENTS`, `$0..$N` positional substitution, `${CLAUDE_SESSION_ID}`
  substitution
- Auto-compaction preserves invoked skills across summarization

**Deprecation status (from `#37447`):**
- Issue is OPEN, labeled `area:skills`, `enhancement`, `stale`.
- 0 comments, no Anthropic assignee, no public timeline.
- Commands keep working indefinitely today, but no project should
  bet on the status quo being permanent.

**Plugin agent frontmatter supported fields (plugins-reference):**
`name`, `description`, `model`, `effort`, `maxTurns`, `tools`,
`disallowedTools`, `skills`, `memory`, `background`, `isolation`.
`agents/auditor.md` uses `color` and `whenToUse`; neither is in the
supported list. Unknown keys are ignored, not errored — this is
cosmetic debt (Finding A5 / Task 8), not a bug.

**How it changed the assessment:**
- Promoted "architecture is fine" → "architecture is on the wrong
  side of a public direction change". This became Finding S1 / Task
  1 — the anchor of the plan.
- Clarified that migration is non-trivial but non-breaking: plugin
  skills resolve to `/pm:<name>` via `<plugin>:<skill-name>`
  namespacing, so user-facing slash commands do NOT change.
- Surfaced bundled scripts via `${CLAUDE_SKILL_DIR}` as a
  prerequisite for Finding S2 / Task 4.

### What they do better than us (current run — new findings only)

- **ccpm** bundles 14 deterministic bash scripts (`standup.sh`,
  `status.sh`, `search.sh`, `next.sh`, `blocked.sh`, `epic-list.sh`,
  `validate.sh`, ...) run at zero LLM token cost. Our
  `/pm:session-end` Phase 1 drift scan is pure LLM work. **Finding
  S2 / Task 4.**
- **ccpm** activates via natural language ("I want to build X",
  "what's our standup", "what's blocked"). Our session lifecycle is
  slash-command-only. This is only reachable via the skills
  migration — **Finding S1 / Task 1.**
- **ccpm** migrated off `/pm:*` slash commands to an Agent Skill.
  Direct A/B evidence that a PM tool on Claude Code should be a
  skill, not a slash command surface. Strongest input into Finding
  S1.

### What we do better than them (unchanged from prior audit)

- **vs ccpm**: lower ceremony. No PRDs, no epics, no GitHub Issues
  requirement, no mandatory parallelism. Solo-author-friendly. We
  deliberately don't compete on the ccpm workflow.
- **vs claude-sessions**: PRESENT / FUTURE trackers, PM-LOG,
  audit command, `deep-research`, native Task integration, session
  drift scan at session-end.
- **vs everything else**: `deep-research` skill integration has no
  direct competitor. Remains a differentiator.

### ccpm naming-collision note (Finding S3 / Task 7)

ccpm v1 shipped `/pm:*`, `/context:*`, `/testing:*`, `/re-init`,
`/code-rabbit`, `/prompt`. A solo developer who remembers the
`/pm:*` surface from ccpm v1 could arrive here confused — the
prefix is identical but the workflow is totally different. ccpm
has since removed those commands, so it's not a live collision,
but a one-line disambiguation in `commands/pm.md` is cheap
insurance. Low priority, small effort.
