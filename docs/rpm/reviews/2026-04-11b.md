## Phase 2: Inward Research

### Hook events (CRITICAL UPDATE)

Since 2026-04-10 and 2026-04-11 morning audits, code.claude.com
documents these new/expanded events:

- **`SessionEnd`** — fires on session termination. Matchers:
  `clear`, `resume`, `logout`, `prompt_input_exit`,
  `bypass_permissions_disabled`, `other`. Direct fit for rpm's
  unwrapped-session detection (see Task 8).
- **`TaskCreated` / `TaskCompleted`** — fires when the native
  `TaskCreate` tool is used. Could sync with tasks.org.
- **`FileChanged`** — fires when a watched file changes on disk.
  Matcher is a literal filename pattern. Could watch `docs/rpm/*`
  for out-of-session edits.
- **`InstructionsLoaded`** — fires when CLAUDE.md / `.claude/rules/*`
  loads. Could warn on stale CLAUDE.md.
- **`CwdChanged`** — fires on working-directory change.
- **`PostToolUseFailure`** / **`StopFailure`** — error-path
  counterparts to existing events.
- **`PermissionDenied`** — auto-mode denial events.

All are valid + supported. None are deprecated. The previously-
announced `once: true` handler option is still supported.

### Skill frontmatter (new fields)

New since last audit:

- **`effort`** — `low` / `medium` / `high` / `max`. Per-skill
  override of session effort level. Match to skill cost:
  `audit` + `deep-research` deserve `high`; lightweight skills
  could go `low`.
- **`context: fork`** — run skill in a forked subagent context,
  preserving main session context. Uses **`agent`** field to pick
  subagent type. Candidates: `deep-research`, `audit project`.
- **Skill-scoped `hooks`** — hooks scoped to specific skill
  lifecycle. Could isolate learning-capture to only fire during
  active rpm skill use.
- **`paths`** — glob activation filter. Not relevant to rpm
  (all skills global), but useful for future specialized work.
- **`shell`** — bash vs powershell for injected commands.

All rpm skills remain valid under current schema. `bootstrap` and
`audit` correctly set `disable-model-invocation: true`; `rpm`,
`session-end`, `tasks`, and `deep-research` correctly default to
model-invocable.

### `$CLAUDE_PLUGIN_DATA`

Still available. rpm uses `docs/rpm/~rpm-*` files by deliberate
choice (git-visible, project-local). Migration is a tradeoff —
rejected (see Deferred).

## Phase 3: Outward Research

### Competitors analyzed (direct competition focus)

| Tool | What it does | Better than rpm | rpm does better | Verdict |
|------|-------------|-----------------|-----------------|---------|
| **claude-code-session-kit** (shihchengwei-lab) | 4 hooks + CLAUDE.md templates. Context monitoring (40/60/70% PostToolUse), handoff validation (Stop), 3-tier CLAUDE.md templates. 92-session tested. | **Context-pressure monitoring with hard-stop gate**; **handoff validator** pre-end-of-session; tiered CLAUDE.md templates | Audit system, doc drift scanning, task tracking, score-based findings, deep-research skill, consolidated session lifecycle | **Adopt context monitoring + SessionEnd validation** |
| **deemkeen/claude-session-management** | `/session-snapshot` save/load/delete + PII sanitization + git-versioned snapshots | Snapshot/rollback model, privacy-first (removes usernames/hostnames/paths) | Tracking + drift + audits + hooks + tasks | Complementary |
| **iannuttall/claude-sessions** | `/project:session-start/update/end/current/list` commands writing markdown session logs | Older, simpler, session log as sole artifact | Everything else | rpm supersedes |
| **claude-session-tracker** (npm) | Auto-logs every conversation to GitHub Issues + Projects board | External archival + GitHub board visualization | Git-native, zero deps | Complementary |
| **claude-plugins-official** (Anthropic) | Official marketplace, 101 curated plugins. External submission via form | Curation + visibility + install flow | rpm is a submission target, not a competitor | **Submit post-2.4.0** |

**Key takeaways:**

1. **The niche is narrower now.** claude-code-session-kit lands
   squarely on rpm's territory (session lifecycle + CLAUDE.md +
   hooks). The two do not overlap on audit scoring or task
   tracking — rpm is still broader — but session-kit's
   context-pressure monitoring and handoff validation are
   directly-addressable gaps.
2. **Hard-stop gates are a proven pattern.** session-kit's 70%
   hard stop is credited in its README with "saved work multiple
   times." rpm's PreCompact checkpoint is downstream of this —
   useful but reactive. Catching context pressure earlier
   (40/60%) is proactive.
3. **Handoff validators are low-effort / high-value.** A simple
   Stop hook that verifies session-state.md has the required
   sections is ~30 lines of bash. rpm's `/session-end` writes but
   doesn't verify. Could add without restructuring.
4. **Zero Python / zero dependencies remains rpm's edge.** Against
   claude-mpm (Python, 60+ services) or claude-session-tracker
   (npm), rpm's "pure bash + markdown" is a differentiator for solo
   devs who distrust install-heavy tooling.
5. **Official marketplace is ready for submission.** 101 plugins
   curated. External submission form exists. After 2.4.0 cuts,
   submit.

## Phase 4: Analysis

### 1. Process Health

Two process failures this session cycle:
- Session 4 ended without `/session-end` — two files uncommitted,
  daily log incomplete. This is the exact failure mode rpm exists
  to detect, and rpm did not detect it because no SessionEnd hook
  exists (Task 8).
- Version bump not automatic. `plugin/CLAUDE.md` says "After
  meaningful changes, bump version" but 5+ feature commits have
  landed without a bump (Task 5). Not hook-enforced.

### 2. Architecture & Code Health

Clean. No legacy directories. 6 skills with valid frontmatter.
scan.sh + session-start-auto.sh bundle deterministic work
appropriately. The Session 3 rearrangement (time-nudge removal →
context-aware suggestion) simplified the hook surface from 5 to 4
hooks without losing functionality. Good.

### 3. LLM Workflow

- Project CLAUDE.md has drifted (Task 1 — ghost hook + missing
  skill) — the worst finding this audit.
- plugin/README.md has drifted the same way (Task 4 — "Five
  hooks").
- context.md has three small formatting drifts (Task 9).
- scan.sh recurrently emits 2 false positives (Task 7).
- Memory codification is strong (10/12 fully CODIFIED).

### 4. Risk & Compliance

No risk issues. All hooks are functional. No security concerns.
The unwrapped Session 4 is a session-discipline failure, not a
data-loss event (working tree is intact).

### 5. Strategic Direction

Strong. The time-nudge removal + context-aware session-end via
context.md rule is a net simplification — smarter behavior with
fewer moving parts. The post-clear continuity feature (`next:`
field) closes a real gap. All signs of healthy iteration.

### 6. Session Discipline

The headline issue this audit. Session 4 demonstrates that
`/session-end` as opt-in is insufficient — the session ended
without it and left drift. Task 8 (SessionEnd hook) is the
mechanical fix. Task 6 (context-pressure monitoring) is the
preventive fix (force `/session-end` before context pressure
causes it to be skipped).

### 7. Competitive Gaps

Two new-since-last-audit competitor patterns worth adopting:
- **Context-pressure monitoring** (session-kit, Task 6)
- **Handoff validation** (session-kit; partially subsumed by
  Task 8's SessionEnd hook)

Two not worth adopting:
- Tiered CLAUDE.md templates (session-kit) — adds surface, low
  value
- Harness-agnostic agentskills.io (ccpm) — sacrifices the
  Claude-Code-native differentiator

### 8. Gaps & Opportunities

Looking at operations scattered across components that would
benefit from formal inclusion:

- **SessionEnd hook (Task 8)** — Claude Code's new event lets rpm
  detect unwrapped sessions. Currently rpm has no detection
  surface for this failure mode; Session 4 is the evidence.
- **Context-pressure monitoring (Task 6)** — Three thresholds
  (soft/hard). The hard threshold can enforce `/session-end`
  before compaction eats detail.
- **Version bump enforcement** — No hook, but could be added
  (PostToolUse on git commit? InstructionsLoaded to warn on
  startup?). Not scored as a finding because the rule is
  sufficiently codified; just needs the human to follow it.
- **TaskCreated/TaskCompleted sync to tasks.org** — The native
  TaskCreate tool creates tasks but rpm's tasks.org is the
  durable backlog. A simple hook could mirror creations. Deferred
  until friction appears — right now `/session-end` does the
  reconciliation.
