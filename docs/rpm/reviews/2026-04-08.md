## Phase 2: Research

Minimum 2 dimensions. Both against authoritative Anthropic docs.

### Dim 1 — Plugin manifest & install flow

**Sources:**
- [Plugins reference — code.claude.com](https://code.claude.com/docs/en/plugins-reference)
- [Discover and install prebuilt plugins — code.claude.com](https://code.claude.com/docs/en/discover-plugins)

**Findings:**
- Canonical manifest location: `.claude-plugin/plugin.json`. A
  root-level `plugin.json` is **not** part of the schema.
- Only `name` is strictly required. Recommended metadata: `version`,
  `description`, `author`, `homepage`, `repository`, `license`, `keywords`.
- Version priority: if set in both `plugin.json` and the marketplace
  entry, `plugin.json` wins.
- Install flow: `/plugin marketplace add <src>` then
  `/plugin install <name>@<marketplace>` (interactive), or
  `claude plugin install <name>@<marketplace>` (CLI).
- `claude --plugin-dir` is a dev/session-scoped install. Fine for local
  iteration, wrong for user-facing "how to install" docs.

**How it changed the assessment:** Promoted the README install fix from
"maybe stale" to High severity. Added a Medium finding for the
duplicate plugin.json mirror (confirmed unnecessary).

### Dim 2 — Plugin subagent invocation

**Sources:**
- [Create custom subagents — code.claude.com](https://code.claude.com/docs/en/sub-agents)
- [Plugins reference §Agents — code.claude.com](https://code.claude.com/docs/en/plugins-reference)

**Findings:**
- Plugin-shipped subagents are namespaced `<plugin-name>:<agent-name>`.
- They are invoked via `subagent_type: "<plugin>:<agent>"` from the
  `Task` / `Agent` tool, or surfaced as `@plugin:agent` mentions.
- Confirmed empirically: this session has `pm:auditor` in its agent
  list — the agent is discoverable and callable right now.

**How it changed the assessment:** Confirmed the dead-subagent finding
with High confidence. The fix is purely to reference the existing
named agent by its `pm:auditor` identifier and de-duplicate the scan
spec.

## Phase 3: Outward Research (competitive analysis)

Added after the user flagged that the original Phase 2 was purely
inward. Researched real alternatives in the Claude Code PM /
session-lifecycle / doc-audit space.

### Competitors discovered

| Alternative | Core concept | Source |
|---|---|---|
| **`claude-sessions`** (iannuttall) | Six slash commands: start / update / current / list / end / help. Per-session timestamped markdown files in `sessions/`. No tracker, no audit. | github.com/iannuttall/claude-sessions |
| **Superpowers** (obra / Jesse Vincent) | 14+ composable skills framework. Enforced brainstorming before coding, TDD, systematic debugging, subagent-driven dev. Methodology framework, not a PM plugin. | github.com/obra/superpowers |
| **Cline Memory Bank** | Six-file structured memory: `projectbrief`, `productContext`, `activeContext`, `systemPatterns`, `techContext`, `progress`. Read at every session start ("not optional"). | docs.cline.bot/prompting/cline-memory-bank |
| **Docs Guardian** (xiaolai marketplace) | Doc quality + freshness enforcer with staleness detection, accuracy checking, and auto-generation via hooks. | github.com/xiaolai/claude-plugin-marketplace |
| **Claude Code native Tasks** (Jan 2025) | Native `TaskCreate / TaskUpdate / TaskList`. Persistent tasks in `~/.claude/tasks/` with dependencies, blockers, multi-session collaboration, `activeForm` spinner display. | code.claude.com changelog |
| **GitHub Spec Kit / AWS Kiro** | Spec-driven development (Specify → Plan → Tasks → Implement). Heavy up-front spec. ThoughtWorks Technology Radar places SDD in "Assess" ring with antipattern warnings. | github.com/github/spec-kit |

### What they do better than us

- **`claude-sessions`** has a mid-session `session-update` command —
  we don't. Long sessions drift between start and end.
- **Claude Code native Tasks** give persistent cross-session task
  state with deps/blockers for free. We reinvent this poorly in
  `FUTURE.org`.
- **Cline Memory Bank** has richer separation of concerns across
  files (product/tech/active/patterns). Better for large projects.
- **Superpowers** enforces methodology (brainstorm-first, TDD, code
  review) as active skills, not just passive guardrails.
- **Docs Guardian** does hook-based continuous freshness sync.
  Ours is on-demand.

### What we do better than them

- **vs `claude-sessions`**: we have PRESENT / FUTURE trackers, PM-LOG
  history, audit command, deep-research integration. They have
  session logs and nothing else.
- **vs Superpowers**: explicit session state + audit history — they
  have none. Superpowers is a methodology layer, not a tracker.
- **vs Cline Memory Bank**: lighter touch; relies on CLAUDE.md
  auto-loading; less maintenance cost for small/medium projects.
- **vs Docs Guardian**: more general audit (process, architecture,
  LLM workflow) vs their doc-only focus. Risk of auto-generated doc
  drift is avoided.
- **vs Spec Kit / Kiro**: lower ceremony; doesn't assume spec-first.
  User has explicitly rejected SDD ceremony.

### Verdicts

| Competitor feature | Verdict | Why |
|---|---|---|
| `session-update` mid-session command | **Adopt** (Finding 6) | Direct gap. Cheap to build. |
| Native Task integration | **Adopt** (Finding 7) | Free functionality we currently ignore. |
| Outward phase in audit (meta) | **Adopt** (Finding 8) | Fixes the exact failure this audit hit. |
| Cline 6-file memory split | **Reject** | Violates "less is more" principle. Our philosophy prefers fewer files with clearer separation via CLAUDE.md. |
| Superpowers methodology layer | **Reject** | Out of scope for a PM plugin. `pm` tracks work; `superpowers` teaches methodology. Complementary. |
| Docs Guardian auto-generation | **Reject** | Risky conflict with hand-authored content. Our audit-on-demand model is deliberate. |
| Spec Kit / Kiro SDD workflow | **Reject** | User rejected spec ceremony. ThoughtWorks flagged SDD antipatterns. |
| deep-research skill integration | **Keep unique** | No direct competitor found. Differentiator. |
